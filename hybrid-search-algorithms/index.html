The Search Landscape

Three paradigms. Three superpowers. One goal: relevance.

Keyword Search

Matches exact terms using inverted indexes. Fast, explainable, great for precise terminology.

  • BM25 scoring
  • Exact phrase matching
  • Facets & filters
  • Lightning fast at scale
Solr · Elasticsearch · Azure Search
Vector Search

Encodes meaning into dense vectors. Understands synonyms, intent, and conceptual similarity.

  • Embedding models (BERT, E5, OpenAI)
  • Approximate Nearest Neighbor
  • Cross-lingual & multimodal
  • No keyword required
FAISS · Pinecone · Weaviate · Milvus
Hybrid Search

Combines both result sets. Leverages lexical precision + semantic recall in a single ranking.

  • RRF / Linear combination
  • Best of both worlds
  • Higher NDCG@10
  • State-of-the-art RAG retrieval
Azure AI Search · Vespa · Weaviate

The verdict: Pure keyword search fails on synonymy and paraphrasing. Pure vector search misses exact terminology and rare entities. Hybrid search is now the default for production RAG and enterprise search.

Lexical Retrieval

Keyword Search: The Immutable Foundation

Inverted indexes map terms to documents. Scoring models rank by term frequency and document rarity.

  • Inverted Index: token → list of (docID, positions, freq).
  • TF-IDF: Weights by term frequency and inverse document frequency.
  • BM25: Probabilistic model with saturation and document length normalization.
  • Phrase queries, wildcards, regex: still best handled lexically.
TF-IDF
wt,d = tft,d × log(N / dft)
BM25
score(D,Q) = Σ IDF(qi) · [ f(qi,D)·(k1+1) ] / [ f(qi,D) + k1·(1-b+b·|D|/avgdl) ]
Semantic Retrieval

Vector Search: Meaning as Geometry

Text (or images, audio, code) is projected into a dense vector space where semantic similarity equals geometric proximity.

  • Embedding model: Bi-encoder (BERT, E5, OpenAI text-embedding-3) maps text → Rd.
  • Similarity metric: cosine, dot product, or Euclidean distance.
  • ANN index: HNSW, IVF, PQ, or graph-based for sub-linear retrieval.
  • Query vector: same model embeds the query; nearest neighbors = top results.

The Gap: Why Neither Is Enough

Real-world queries contain exact entities, synonyms, abbreviations, and intent — all at once.

Scenario Keyword Search Vector Search
Exact product SKU✅ Perfect match❌ May miss exact ID
Synonymy (“auto” vs “car”)❌ Needs synonym map✅ Naturally handles it
Conceptual question❌ Poor on concepts✅ Strong semantic match
Rare / OOV terms✅ Matches if in index❌ Depends on model vocabulary
Explainability✅ Highlighted terms❌ Black-box distance
Multilingual❌ Needs translation✅ Cross-lingual by design

Insight: The weaknesses of keyword search are the strengths of vector search — and vice versa. A production system needs both signals fused intelligently.

Parallel Retrieval

Query is dispatched to both the inverted index and the vector index simultaneously. Each returns top-k candidates independently.

Candidate Pool

The union of both sets forms a candidate pool. Overlapping documents get a boost from appearing in both lists.

Fusion Function

A scoring function re-ranks the pool. Options: linear combination of scores, RRF, learned models, or two-stage rerankers.

Fusion Strategies

How do you combine two ranked lists into one? Three dominant approaches.

Linear Combination
Shybrid = α · Skeyword + β · Svector

Requires normalized scores. α and β are tuned on validation data. Simple and fast.

Best when both scores are calibrated and you have training data.
Reciprocal Rank Fusion (RRF)
RRF(d) = Σ 1 / (k + ri(d))

Rank-based, score-agnostic. k is a constant (usually 60). Robust across different scoring scales.

Best when scores are not comparable or you want zero hyperparameter tuning.
Learned Fusion
S = MLP( [Skw, Svec, feats] )

Train a small model (LambdaMART, neural ranker) on click / relevance labels. Most powerful but data-hungry.

Best at web scale with abundant behavioral data.

Recommendation: Start with RRF — it requires no score normalization and is surprisingly strong. Move to linear combination if you have calibrated scores and a dev set. Use learned fusion only at massive scale.

Azure AI Search: Architecture Overview

A managed cloud search service with native support for keyword, vector, and hybrid retrieval — plus semantic reranking and AI enrichment.

Inverted Index

Traditional full-text with analyzers, tokenizers, synonyms, and BM25 scoring. Supports facets, filters, and geospatial.

Vector Index

HNSW-based approximate nearest neighbor. Configurable m (connections) and efConstruction for speed/recall trade-off.

Semantic Ranker

Microsoft's transformer-based reranker (BERT-class) that reorders top results using deep semantic understanding.

Azure AI Search: The Details

Key features that make it a production-grade hybrid search platform.

Integrated Vectorization

Vectorizer support at query time means you send raw text; Azure calls the embedding model (OpenAI, Azure OpenAI, or custom) automatically. No client-side embedding needed.

Hybrid + Semantic Reranking

Set queryType=semantic and searchMode=all to get the full stack: BM25 + vectors + semantic captions + semantic answers.

AI Enrichment (Skillsets)

During indexing, built-in skills extract entities, detect language, run OCR, generate image captions, and chunk documents. Outputs become searchable fields.

Scoring Profiles & Boosting

Assign weights to fields, use freshness functions, geospatial distance, and tag boosting. Combine with vector similarity for business-aware ranking.

Facets, Filters & Order-By

$filter, $orderby, $facet parameters work across both lexical and vector queries. Essential for e-commerce and catalog search.

Multi-Vector Fields

One document can have multiple vector fields (title vector, description vector, image vector). Query any or all simultaneously with different weights.

Query Parsing

Simple (all terms), full (Lucene syntax), or semantic. Simple is fastest; semantic adds ~100-200ms for reranking.

Vector Query Payload

You pass the raw vector (or use a vectorizer) with k, fields, and exhaustiveness parameters.

Exhaustive vs ANN

Set exhaustive=true for exact k-NN at higher latency. Default HNSW is approximate and much faster.

Two-Stage Retrieval

Re-ranking & Cross-Encoders

First stage retrieves candidates fast. Second stage scores query-document pairs with a powerful model.

  • Bi-Encoder (retrieval): Fast, independent embeddings. Good recall, weaker precision.
  • Cross-Encoder (reranking): Query + doc concatenated. Full cross-attention. Much more accurate.
  • Azure Semantic Ranker: Microsoft's managed cross-encoder. No infra to run.
  • MonoT5 / BERT-reranker: Open-source alternatives for self-hosted stacks.

Chunking Strategy

Fixed-size sliding windows, semantic chunking, or hierarchical parent-child. Chunk size and overlap directly affect retrieval quality.

Embedding Model Choice

OpenAI text-embedding-3-large, E5-multilingual, Cohere Embed, or fine-tuned domain models. MTEB leaderboard is your guide.

Index Tuning

HNSW parameters: m (16-64), efConstruction (100-500). Higher = better recall, slower build. efSearch at query time controls latency/recall.

Architect's rule: If your users ask questions in natural language and use specific IDs, SKUs, or names, you need hybrid. If your corpus is entirely unstructured text, pure vector may suffice. If every query is an exact ID lookup, stick to keyword.

Key Takeaways

The modern search stack in six bullets.

1. Keyword is not dead

BM25 remains the best tool for exact matches, rare terms, and explainable retrieval. It is fast, cheap, and robust.

2. Vectors capture meaning

Dense embeddings solve synonymy, paraphrasing, and conceptual queries that lexical search cannot touch.

3. Hybrid is the default

Run both in parallel and fuse with RRF or linear combination. The overlap between lexical and semantic hits is a strong relevance signal.

4. Reranking wins

A cross-encoder or semantic ranker on top-100 candidates lifts NDCG significantly without destroying latency.

5. Azure AI Search is full-stack

Native BM25 + HNSW vector + semantic reranker + skillsets + filters. A single API call can do it all.

6. Chunking & embeddings matter

A great retrieval engine cannot save bad chunks or weak embeddings. Invest in preprocessing and model selection.
