The Problem

Three fraud patterns hide in unstructured bidding documents

🤝

Supplier Bid Rigging

Criticality: Medium
  • Same supplier group bids together repeatedly
  • A supplier always participates but never wins
  • Fixed groupings take turns winning across plants
💰

Abnormal Supplier Pricing

Criticality: High
  • Fixed proportional relationships between rivals' line items
  • Excessive total-price gaps (>20% low-to-high)
📄

Quotation-File Anomalies

Criticality: High
  • Author = final editor on a "competitor's" quote
  • File modification times clustered in a one-hour window

All three live across the same per-bid evidence: a requirement, a bid summary, vendor quotation PDFs, and quotation emails — none of it in a structured database.

Architecture Options

How do we honor the China boundary? — Three patterns

Option A

Twin-Stack

Two fully independent end-to-end builds. Nothing crosses the border. Maximum isolation, maximum duplication.

Option B · Recommended

Federated Hub-and-Spoke

One shared design; data & LLM stay in-region. Only non-sensitive aggregates roll up to a global view. One-way, north-bound only.

Option C

Global-Only + Redaction

Single global stack; China data de-identified at the edge before processing. Cheapest — but the residency risk you chose to avoid.

All three share the same reference architecture internally. They differ only in how much is duplicated and what — if anything — crosses the border.

🔁 Two ingestion modes

  • Backfill — one-time sweep of 2023→today (~60k bids) via Azure OpenAI Batch API (cheaper, quota-friendly), throttled, idempotent, resumable, with a one-time cost estimate.
  • Daily delta — Graph change/webhook picks up only new & modified bid folders (~80/day).

⚠️ Why embedded, not SharePoint

SharePoint's Author/Editor is just who uploaded the file — usually one buyer for the whole folder, so it would flag every bid. The real signal is the document's own embedded metadata; we degrade gracefully where files were flattened or scrubbed.

Layer 2 · Extraction

Turn unstructured evidence into a canonical bid record

Bid { bid_id, plant, category, requirement_text, award: { winner_vendor_id, awarded_total, date }, vendors: [{ vendor_id, // resolved, not raw name raw_name, source_files[], quote_total, currency, line_items: [{ sku, qty, unit_price }], email: { sender, sent_ts, body_hash } }], file_meta: [{ file, author, last_editor, created_ts, modified_ts }] }

The model emits strict JSON against this schema; a validator rejects/repairs malformed output before it lands.

  • LLM structured extraction — frontier model reads each quote/email and fills the schema, normalizing currencies & units.
  • Entity resolution (first-class) — a persistent vendor registry maps every alias, email, tax-ID & filename fragment to one vendor_id via deterministic keys + LLM-assisted matching, with a compliance merge/split review UI. Without it the rigging graph is noise.
  • Provenance — every field keeps a pointer to its source file + page, so findings are always citable.
  • Embeddings — requirement & quote text are vectorized into the RAG index for the chat layer.
Layer 3 · Detection Engine

Deterministic rules + LLM reasoning

📐 Rules (cheap, explainable)

Thresholds & graph queries over the canonical records — price gaps, co-bidding frequency, win/loss streaks, metadata matches. Fast, deterministic, auditable. Runs over the whole corpus nightly.

🧠 LLM (judgment, context)

For borderline cases the model weighs context — reads the actual quotes/emails, explains why a pattern looks like collusion, and drafts the analyst-ready narrative. Invoked selectively to control cost.

Rules do the broad sweep across 20k bids; the LLM adds judgment on the few hundred that trip a threshold. Next three slides: one detector per fraud pattern.

Detector 1 · Bid Rigging

Collusion is a graph problem across bids

🔗
Repeated co-bidding

Same supplier set appears together across many tenders.

co_bid_count(A,B) ≥ N over window
🎭
Perennial loser

A vendor bids often but never wins over an extended period — a likely "cover" bidder.

participations ≥ N AND wins = 0
🔄
Rotation / bid-sharing

A fixed group takes turns winning across projects & plants.

win-share evenly cycled within group

How it works

  • Build a supplier co-occurrence graph — nodes = vendors, edges weighted by shared tenders.
  • Detect dense clusters + analyze win rotation within them.
  • LLM explains the suspicious cluster in plain language for the case file.

Entity resolution from Layer 2 is the enabler — without one stable vendor_id, the graph is noise.

Detector 2 · Abnormal Pricing

When "competing" quotes move together

📈
Fixed proportional relationships

Line items from different vendors hold a constant ratio — a fingerprint of coordinated pricing.

corr(vendorA_items, vendorB_items) ≈ const ratio
↔️
Excessive total-price gap

Lowest vs highest total within one tender is implausibly wide — a sign of token "filler" bids.

(max_total − min_total) / min_total > 20%

How it works

  • Align line items across vendors within the same tender (the bid folder gives us this grouping for free).
  • Compute pairwise correlation & ratio stability per SKU.
  • Flag fixed ratios and outlier total gaps; LLM reviews the line-item table and confirms/dismisses.

Thresholds (the 20% gap, ratio tolerance) are config in the global control plane — tunable without code.

Detector 3 · Quotation-File Anomalies

The metadata gives the collusion away

✍️
Author = final editor across "rivals"

One person authored/edited quotes attributed to different vendors.

author(file_i) == editor(file_j), i≠j vendor
⏱️
Clustered modification times

Independent vendors' files last modified inside a tight window.

max(modified_ts) − min(modified_ts) < 1 hour

How it works

  • Pure deterministic rules over file_meta — fast, fully explainable, no LLM needed.
  • Cross-reference author/editor identities against the vendor each file belongs to.
  • Cluster modification timestamps per tender.

Scope reminder: keyed off the document's embedded properties (not the SharePoint uploader identity). High-signal where present; degrades gracefully where files were flattened/scrubbed on upload.

Weighted, not black-box

score = Σ wᵢ·sᵢ over fired signals; tier by threshold. Every weight and contribution is shown.

bid-abc = 0.4·cobid + 0.4·ratio + 0.2·meta
= 0.85 → HIGH

Tier drives action

High → auto-opens a case & alerts. Medium → review queue. Low → logged for trend tracking.

Always explainable

Every score ships with its reasons + citations — compliance never sees a bare number.

Evaluation

No eval, no trust — prove the detectors

  • Labeled golden set — compliance tags historical bids: known-collusion, known-clean, and known-correct extraction values.
  • Eval harness scores every layer: extraction accuracy, entity-resolution precision/recall, per-detector precision/recall, chat grounding.
  • Gate on change — re-runs on every rule, prompt, or threshold edit, so the 20% gap & co-bid counts get tuned with evidence, not guesses.
  • Reported to compliance — a standing false-positive rate per detector. The number that earns (or loses) their trust.
