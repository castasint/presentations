Why this matters

The N×M integration problem

Every agent that needs M tools, must talk to N peer agents across K UIs creates a combinatorial explosion of bespoke, brittle glue code.

  • Custom connectors rot the moment an API changes
  • No agent can discover or trust another vendor's agent
  • UIs are hard-wired to one model's streaming format
  • Governance & audit scatter across a dozen places

Protocols collapse N×M custom integrations into N+M standard ones — the shift HTTP, USB-C and TCP/IP each delivered.

N×M

without standards — bespoke, fragile, unmonitored

N+M

with a shared protocol layer — reusable & governed

The mental model

Each protocol owns one axis. They are complementary, not competing — you will run several at once.

↓ DOWNWARD

Transports & Encodings

Most agent protocols reuse boring, proven web standards on purpose — interoperability beats novelty.

Protocol Encoding Transport
MCP JSON-RPC 2.0 stdio + Streamable HTTP
A2A JSON-RPC 2.0 (+ gRPC, REST) HTTP(S) + SSE
AG-UI JSON events SSE / WebSocket
ACP REST / JSON HTTP
ANP JSON-LD HTTP + DID
No new transport was invented — that's the point: standards ride on HTTP.
Foundations

Streaming Patterns

Long-running agents need more than request/response — protocols choose from three well-understood streaming models.

SSE

Server-Sent Events — one-way server→client stream over a single HTTP connection. Each event is a newline-delimited JSON payload. Used by MCP A2A AG-UI for token streaming and progress events.

WebSocket

Full-duplex persistent connection — both sides send messages freely. Used by AG-UI when the UI needs to push user actions (typing, interrupts) back to the agent while it streams output.

Webhooks / Push

Server calls back a client-supplied URL when a long-running job completes or changes state. Used by A2A push notifications — client disconnects safely; result arrives later.

Foundations

Identity, Auth & Trust

Every agent call crosses a trust boundary — protocols layer standard auth mechanisms rather than invent new ones.

OAuth 2.1
Scoped bearer tokens with short TTLs. Used by MCP remote servers and A2A for delegated agent-to-agent calls.
API Keys / HTTP Auth
Simple static credentials over HTTPS. Low friction for ACP and A2A internal deployments.
mTLS
Mutual TLS — both client and server present certificates. Used for high-assurance service-to-service lanes where token theft is a serious threat.
DIDs / Decentralized Identifiers
Self-sovereign identity anchored on a verifiable data registry. Enables cross-org, trustless agent identity — core to ANP.
Discovery is not authorization. Always scope and verify tokens — knowing an agent exists does not mean you may call it.
Foundations

The Content Model

Agents exchange structured, multi-modal payloads — not raw strings. Each protocol wraps content in typed "parts" so consumers know exactly what they're getting.

  • Structured I/O via JSON Schema — tool inputs and outputs are validated against schemas declared in the manifest; no guessing types at runtime.
  • MCP content blocks — text, image, audio, embedded resource, each with a type discriminator.
  • A2A Parts — TextPart, FilePart (inline bytes or URI + MIME), DataPart (arbitrary JSON).
  • ACP MessagePart — role + list of parts, each with content_type (MIME) so HTTP-native tooling can route and render without extra parsing.
{
  "role": "agent",
  "parts": [
    {
      "type": "text",
      "content": "Here is the chart:"
    },
    {
      "type": "file",
      "content_type": "image/png",
      "uri": "https://store/chart.png"
    },
    {
      "type": "data",
      "content_type": "application/json",
      "data": { "rows": 42 }
    }
  ]
}
Foundations

Sync, Async & Long-Running

Real agent work spans milliseconds to hours — protocols must support all three interaction shapes or agents become toys.

Synchronous

Request → Response in one HTTP round-trip. Best for fast tool calls where latency is <2 s. MCP tool invocations default here. Simple to reason about; fails for anything that might timeout.

Streaming

Incremental output delivered as it's produced — keeps UIs responsive during generation. AG-UI SSE token streams, MCP progress notifications, and A2A task update events all use this shape.

Async Long-Running

Submit → poll / push — task survives client disconnects. Critical for minutes-to-hours jobs: data pipelines, multi-step research, code review. A2A Tasks + push notifications and ACP async runs address this directly.

MCP — Model Context Protocol
The resource layer. One model, any tool or data source. The protocol that started the movement.
MCP

What is MCP?

Anthropic · Nov 2024
open standard · JSON-RPC 2.0

Model Context Protocol is an open standard that defines how applications provide tools, data, and context to LLMs — a universal adapter layer between models and the world.

USB-C for AI: Just as USB-C standardizes how devices connect to peripherals, MCP standardizes how any LLM application connects to any tool or data source — plug in once, works everywhere.
Created by
Anthropic — open-sourced November 2024
Wire format
JSON-RPC 2.0 over stateful sessions
Goal
Replace N×M bespoke integrations with a single shared protocol
MOTIVATION

The problem MCP solves

Before MCP

Every model × every tool = custom glue code

  • N models × M tools = N×M integrations
  • Each integration bespoke & brittle
  • No shared auth, schema, or lifecycle
  • Duplicated effort across teams & orgs
  • Context silos — data locked away

After MCP

One standard: N models + M servers = everything works

  • N + M implementations, not N × M
  • Standard lifecycle, negotiation, schema
  • Shared OAuth 2.1 authorization model
  • Write a server once; every host uses it
  • Resources, tools, prompts — unified API
ARCHITECTURE

Host, Client, Server

Three participants form every MCP deployment. One host runs many clients; each client holds exactly one stateful connection to one server.

HOST
LLM app · manages model + clients
⇄spawns / connects
CLIENT ×N
1:1 stateful session per server
⇄JSON-RPC 2.0
SERVER ×N
Lightweight capability provider
Examples: Host = Claude Desktop, VS Code, an agent runtime. Server = GitHub MCP server, Postgres MCP server, a custom internal API adapter.
PARTICIPANTS

The three participants

Host

The LLM application or agent runtime. Manages the model inference, creates clients, enforces security policy, and presents results to the user.

e.g. Claude Desktop, Cursor, a custom agent runtime

Client

Lives inside the host. Maintains a single, persistent, stateful JSON-RPC session with exactly one MCP server. Translates model intent into protocol calls.

One client per server; many clients per host

Server

A lightweight program that exposes capabilities — tools, resources, prompts — to the host and model. Can be local or remote; stateless or stateful.

e.g. GitHub, Slack, Postgres, custom API wrappers

WIRE FORMAT

Built on JSON-RPC 2.0

Every MCP message is a standard JSON-RPC 2.0 envelope. Sessions are stateful — both sides track connection state, negotiate features, and exchange notifications without request/response pairing.

Requests
id + method + params → expects a response
Responses
Matching id + result or error
Notifications
No id — fire-and-forget, no reply expected
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "search_orders",
    "arguments": {
      "query": "status:open",
      "limit": 20
    }
  }
}

Generic JSON-RPC 2.0 request envelope

LIFECYCLE

Connection lifecycle

Every MCP session follows a strict, predictable lifecycle — ensuring both sides agree on capabilities before any work is done.

  1. initialize — Client sends protocolVersion + its capability declarations; server responds with its own capabilities and chosen protocol version.
  2. Capability negotiation — Both sides inspect what the other supports; fall back gracefully to the intersection of features.
  3. initialized notification — Client fires a one-way notification confirming it is ready; operation phase begins.
  4. Operation — Free exchange: tools/call, resources/read, prompts/get, sampling/createMessage, etc.
  5. Shutdown — Either side can terminate cleanly; stdio servers exit, HTTP sessions close.
Note: No work may be performed before the initialized notification is sent by the client.
NEGOTIATION

Capability negotiation

During initialize, both sides declare the optional features they support. Neither side may rely on a capability the other did not advertise.

Client declares

  • sampling — host can run LLM completions for servers
  • roots — host exposes filesystem boundaries
  • elicitation — host can prompt user for input

Server declares

  • tools — exposes callable functions
  • resources — exposes URI-addressed data + subscribe
  • prompts — exposes reusable prompt templates
  • logging — can emit structured log messages

Features not declared are never attempted — clean forward-compatibility.

SERVER PRIMITIVES

Primitive 1: Tools

Tools are executable functions the model decides to invoke. The model reads descriptions and schemas, picks the right tool, supplies arguments — the host executes, returns results.

  • Model-controlled — model decides when and how to call
  • name — unique identifier within the server
  • description — natural-language guide for the model
  • inputSchema — JSON Schema for arguments (validated)
  • outputSchema — optional structured output schema
  • Annotations — readOnlyHint, destructiveHint signal risk level to host/user
  • Methods — tools/list to enumerate, tools/call to invoke
  • Results — content blocks (text, image, embedded resource)
Design tip: Write tool descriptions as if briefing a smart but uninformed colleague. The model only knows what you put in description and inputSchema.

SERVER PRIMITIVES

Primitive 2: Resources

Resources expose read-only, URI-addressed data — documents, database rows, files, API snapshots. The application (host) decides which resources to attach to context, not the model.

  • Application-controlled — host decides what to load
  • URI-addressed — file://, postgres://, https://, custom schemes
  • Read-only — resources are context, not actions
  • Templates — parameterized URI patterns via resources/templates/list
  • Subscribe — host can watch for updates; server pushes resources/updated notifications
  • Methods — resources/list, resources/read, resources/subscribe
Use resources for: Long documents that shouldn't be re-fetched per turn, live database snapshots the host injects into context, configuration files, and any read-only structured data the model needs to reason over.

SERVER PRIMITIVES

Primitive 3: Prompts

Prompts are reusable, parameterized workflow templates exposed by a server — think slash commands. The user picks and invokes them; the server fills arguments and returns a ready-made message sequence.

  • User-controlled — user selects from a menu / slash command
  • Parameterized — server declares argument names + descriptions
  • Returns messages — a ready-to-send array of user/assistant turns
  • Embeds resources — can inline resource content into the prompt
  • Methods — prompts/list, prompts/get
{
  "method": "prompts/get",
  "params": {
    "name": "review_pr",
    "arguments": {
      "pr_number": "42",
      "focus": "security"
    }
  }
}

Server returns a structured message array ready for the model

CLIENT PRIMITIVES

Client power: Sampling

Sampling inverts the flow — a server asks the host's LLM to reason on its behalf. The server gets agentic intelligence without shipping its own model or API keys.

  • Method: sampling/createMessage
  • Server supplies messages + modelPreferences
  • Host picks the actual model and runs inference
  • Result returned to the server as a completion
  • Enables multi-step reasoning inside a server tool
Human approval recommended — hosts should surface sampling requests to the user before executing, especially for sensitive operations. 
{
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "Classify this log line..."
        }
      }
    ],
    "maxTokens": 256
  }
}

Server → Client: host runs the LLM, returns completion

CLIENT PRIMITIVES

Roots & Elicitation

Roots

The client tells the server which filesystem paths or URI prefixes it is allowed to operate within. Servers must respect these boundaries — no reads or writes outside declared roots.

  • Declared by client during or after init
  • Can be file:// paths or custom URI schemes
  • Server receives updated list via roots/list_changed notification
  • Primary sandbox enforcement mechanism for local servers

Elicitation

Added in the 2025 spec — a server may pause mid-operation and ask the user (via the host) for structured input before proceeding.

  • Server sends a schema describing what it needs
  • Host surfaces a prompt or form to the user
  • User response returned as structured data
  • Enables interactive, human-in-the-loop tool flows
TRANSPORTS

stdio vs Streamable HTTP

MCP defines two standard transports. Choose based on where your server lives.

Dimension stdio (local) Streamable HTTP (remote)
Deployment Local subprocess; host spawns server Remote HTTP endpoint; client POSTs requests
Streaming Synchronous stdin/stdout; no partial streaming Server streams responses via SSE on the same connection
Session model Always stateful — process lifetime = session Supports both stateless (per-request) and stateful modes
Auth OS-level process isolation OAuth 2.1 with audience-bound tokens
Best for Local tools, dev environments, IDE plugins Hosted APIs, multi-tenant SaaS, shared team servers

Streamable HTTP (introduced March 2025) supersedes the older separate HTTP + SSE transport.

SECURITY

Authorization with OAuth 2.1

For HTTP transport, MCP specifies OAuth 2.1. The MCP server acts as an OAuth Resource Server; a separate Authorization Server issues audience-bound tokens.

  1. Client discovers Auth Server via Protected Resource Metadata
  2. Client registers dynamically (Dynamic Client Registration)
  3. User authenticates; Auth Server issues token scoped to resource
  4. RFC 8707 Resource Indicators bind token to the specific MCP server audience
  5. Client presents token in every request; server validates audience + scopes
No static long-lived keys. The spec explicitly discourages API keys shared across servers. Tokens must be audience-bound — a token issued for Server A must not be accepted by Server B.
Token pass-through anti-pattern: Never forward a token you received to a downstream MCP server. Obtain a fresh, scoped token for each server.
SECURITY

The sharp edges

Tool poisoning

Malicious or compromised servers embed hidden instructions inside tool descriptions or results. The model reads them as trusted context and follows attacker directives.

Pin / allow-list server packages. Review tool definitions before enabling.

The lethal trifecta

Any MCP deployment combining all three is high-risk: (1) access to private data, (2) exposure to untrusted content, (3) ability to exfiltrate data.

Audit each server against all three legs. Remove any unnecessary leg.

Confused deputy & token passthrough

A server acts on behalf of the model using elevated host permissions it shouldn't have, or forwards tokens to unauthorized downstream systems.

Scope tokens tightly; never pass through received tokens.

Mitigations: Explicit user consent for each server connection · pin/allow-list server packages by hash · review tool definitions before enabling · scope OAuth tokens to minimum required · sandbox server execution · surface sampling requests to user before executing.

DECISION GUIDE

MCP — when to use

Use MCP when…

  • One model/agent needs grounded access to external tools or data
  • You want governed, auditable tool invocations with consent
  • You need to reuse existing MCP servers (GitHub, DB, Slack…)
  • You want standard auth (OAuth 2.1) without custom glue
  • You need resources (read-only context) injected into the model
  • You want reusable prompt templates surfaced as slash commands

Reach for something else when…

  • Agent-to-agent orchestration — use A2A (Google's protocol for peer agents communicating with each other)
  • Streaming UI to humans — use AG-UI (the agent-to-frontend event stream protocol)
  • You need bidirectional real-time agent collaboration across orgs
  • Your primary need is rendering live UI updates, not tool access
One-line takeaway: MCP is the resource and tool layer — pair it with A2A for multi-agent orchestration and AG-UI for live user-facing interfaces.

A2A — Agent2Agent
The collaboration layer. Opaque agents discover each other and delegate work — across frameworks, clouds and vendors.
A2A

What is A2A?

Google · Apr 2025 → Linux Foundation
HTTP · JSON-RPC 2.0 · SSE

A2A is an open protocol that lets opaque agents — built on different frameworks, by different vendors — discover each other, communicate, and delegate tasks.

  • Launched by Google in April 2025 with 50+ partners (Atlassian, Salesforce, SAP, ServiceNow, Box, PayPal, Cohere…)
  • Donated to the Linux Foundation in June 2025 — now the vendor-neutral A2A Project
  • Current spec: v0.3 — transports include JSON-RPC/HTTP, SSE, and gRPC
The famous framing: "MCP for tools, A2A for agents" — they are complementary, not competing.
THE PROBLEM

Islands of agents

Without A2A

Every agent pair requires bespoke integration

  • Custom glue code per agent pair
  • No shared discovery or capability matching
  • Vendor lock-in — each agent speaks its own protocol
  • N × M integration matrix at scale
  • Re-implement auth, streaming & retries every time

With A2A

Uniform protocol; any agent delegates to any other

  • Publish an Agent Card — other agents discover you
  • Capability matching built into the spec
  • Standard auth, streaming (SSE) & push notifications
  • Cross-vendor, cross-framework by design
  • N + M instead of N × M integrations
DESIGN PRINCIPLES

Six principles of A2A

① Agentic capabilities

Agents collaborate naturally without sharing memory, tools, or internal context — true peer autonomy.

② Existing standards

Built on HTTP, JSON-RPC 2.0, and SSE — no new transport; works with existing infrastructure.

③ Secure by default

Enterprise-grade auth via OAuth2, OIDC, API keys, and bearer tokens declared in the Agent Card.

④ Long-running tasks

Tasks span minutes to hours or days. Push notifications and SSE keep clients in sync — including humans in the loop.

⑤ Modality agnostic

Supports text, audio, video, files, and structured data. No single-modality assumption.

⑥ Opaque execution

Agents never expose internal reasoning, tool names, or model weights — a clean capability boundary.

DISCOVERY

The Agent Card

A public JSON manifest published at https://HOST/.well-known/agent-card.json. Enables discovery and capability matching without any prior agreement.

  • name / description / version / url
  • provider — organization & contact
  • capabilities — streaming, pushNotifications, stateTransitionHistory
  • defaultInputModes / outputModes — modalities
  • skills[] — what the agent can do (id, name, tags, examples)
  • securitySchemes — OAuth2, OIDC, API key, bearer
Discovery flow: client fetches /.well-known/agent-card.json, reads capabilities & skills, selects best-fit agent, negotiates auth — all before sending a single task.
Extended card: agent/getAuthenticatedExtendedCard reveals additional metadata visible only to authenticated callers.

CONTENT MODEL

Messages & Parts

A Message is one turn in the conversation (role: user or agent). It carries one or more Parts — enabling rich multi-modal payloads.

TextPart

type: "text"

Plain or markdown text. The most common part type — prompts, reasoning, status messages.

FilePart

type: "file"

Binary content (base64 bytes) or a URI reference, plus mimeType. For PDFs, images, audio, video.

DataPart

type: "data"

Structured JSON payload — tabular results, form data, API responses. Enables machine-to-machine data exchange.

A single message can combine multiple part types — e.g. TextPart (instruction) + FilePart (PDF) + DataPart (metadata).

OUTPUTS

Artifacts

An Artifact is a discrete, named output that the remote agent produces for a Task. Artifacts are made of Parts — the same TextPart / FilePart / DataPart primitives.

  • Attached to the Task in artifacts[]
  • Each has an artifactId, optional name & description
  • Streamed incrementally via SSE — clients see partial artifacts as they are generated
  • Persists after task completion — client can retrieve via tasks/get
Example artifacts: a PDF report (FilePart), a structured invoice (DataPart), a markdown summary (TextPart), or an audio clip (FilePart with audio MIME type). One task may produce many artifacts.

ASYNC

Long-running tasks & push

When tasks run for minutes to hours — or clients go offline — SSE alone isn't enough. A2A push notifications let the remote agent call the client back.

  1. Client registers a webhook URL via tasks/pushNotificationConfig/set
  2. Client sends the task and disconnects (no open SSE connection needed)
  3. Remote agent works asynchronously — uses its own tools/models
  4. Agent POSTs TaskStatusUpdate events to the webhook as state changes
  5. Client receives update, optionally calls tasks/get or tasks/resubscribe

SECURITY

Auth & enterprise security

The Agent Card declares securitySchemes — clients obtain credentials out-of-band and pass them via standard HTTP auth headers.

  • OAuth2 — authorization code / client credentials flows
  • OpenID Connect — identity federation
  • API Key — header or query parameter
  • HTTP Bearer — JWT or opaque token

The remote agent is treated as a standard opaque enterprise application — no special agent-aware identity layer required.

Discovery ≠ authorization. Finding an Agent Card does not grant access. Auth must be negotiated separately — the card only describes which scheme to use.
auth-required state: if credentials expire mid-task, the agent transitions to auth-required and pauses — the client re-authenticates and resubmits.

DECISION GUIDE

A2A — when to use

Use A2A when…

  • Multiple specialist agents must collaborate across teams or vendors
  • Agents are built on different frameworks or clouds
  • Tasks are long-running (minutes → hours) or need human-in-the-loop
  • You need capability discovery without prior bilateral agreement
  • Delegation should be opaque — internals stay private

Don't use A2A for…

  • Reaching raw APIs, databases, or files — use MCP
  • Streaming tokens to a human UI in real time — use AG-UI
  • Simple in-process tool calls — direct function calls are cheaper
Limitations: opaque agents are hard to debug & trace; cross-org trust and discovery still maturing; spec is young (versioning churn); unbounded fan-out can exhaust budgets without rate controls.

Takeaway: A2A is the horizontal glue between specialist agents — pair it with MCP (vertical tools) for a complete agentic system.

AG-UI & A2UI
the human layer

The last mile. Streaming an agent's work to a frontend, rendering rich UI, and letting a human steer.
THE PROBLEM

The agent → human problem

Without a standard, every UI is hard-wired to one model's streaming format — brittle, bespoke, non-portable.

Without a standard

  • Custom streaming adapter per LLM provider
  • Token format breaks when you swap models
  • Tool-call events differ across frameworks
  • State sync re-invented every project
  • Human-in-the-loop logic scattered in glue code

With AG-UI

  • One typed-event contract for all agents
  • Swap LLMs without touching the frontend
  • Tool execution visible in the UI out of the box
  • Shared state synced via JSON Patch
  • Approval gates and interrupts are first-class
OVERVIEW

What is AG-UI

An open, lightweight, event-based protocol from CopilotKit (2025) that standardises how front-end apps connect to AI agents through a stream of typed JSON events.

vs MCP

MCP wires the agent to tools. AG-UI wires the agent to the user interface.

vs A2A

A2A connects agents to other agents. AG-UI connects agents to humans.

The last mile

MCP → A2A → AG-UI. The final hop from agent runtime to the human surface.

Last mile to the user — AG-UI completes the protocol stack: tool access (MCP), multi-agent coordination (A2A), and now the human interaction layer.
CORE MODEL

The event-stream model

~16 typed JSON events, grouped into five categories. Transport-agnostic: works over SSE, WebSockets, or webhooks.

EVENTS · LIFECYCLE

Events: Lifecycle

Signal when a run or step starts, finishes, or errors — giving the UI a clear execution skeleton.

RUN_STARTED
A new agent run has begun; carries run ID and initial metadata.
RUN_FINISHED
The run completed successfully; final signal to close the stream.
RUN_ERROR
The run failed; payload includes error message and stack context.
STEP_STARTED
A logical step within the run has begun (e.g., planning, retrieval, generation).
STEP_FINISHED
A step completed; UI can mark milestones or progress indicators.
Why it matters — Lifecycle events let any UI render accurate progress bars and error states without agent-specific parsing.
EVENTS · TEXT

Events: Text messages

Three events bracket every streamed response, enabling token-by-token rendering identical to native LLM streaming.

TEXT_MESSAGE_START

Opens a new message; carries a message_id and role. UI allocates a chat bubble.

TEXT_MESSAGE_CONTENT

Token delta — emitted once per token chunk. UI appends the delta field to the bubble in real time.

TEXT_MESSAGE_END

Closes the message. UI finalises the bubble, enables copy/action controls.

Note: no full text in END — reconstruct from deltas.

EVENTS · TOOL CALLS

Events: Tool calls

Four events expose live tool execution — the UI can show exactly what the agent is calling and what came back.

  • TOOL_CALL_START — agent begins invoking a tool; carries tool_call_id and tool_name. UI can show a spinner labelled with the tool name.
  • TOOL_CALL_ARGS — streaming argument deltas (mirrors TEXT_MESSAGE_CONTENT for args). UI can preview the forming JSON in real time.
  • TOOL_CALL_END — argument stream closed; tool is now executing. UI transitions from "building args" to "running".
  • TOOL_CALL_RESULT — tool returned; payload carries the result or error. UI renders the output inline and resumes streaming.
EVENTS · STATE

Events: State management

Shared agent–UI state kept in sync efficiently — full snapshots on connect, incremental patches thereafter.

STATE_SNAPSHOT
Full serialised state object sent once (e.g., on connect or reset). UI replaces its local copy entirely.
STATE_DELTA
Incremental update via JSON Patch (RFC 6902) — only the changed paths are sent. Bandwidth-efficient for large states.
MESSAGES_SNAPSHOT
Full conversation history snapshot. Used to re-hydrate UI after reconnection or page reload.

Why JSON Patch?

RFC 6902 patches express add, remove, replace, move operations on a JSON tree. A large agent state (100 KB) can be updated with a 50-byte patch — no re-serialisation of unchanged data.

Result: agent and frontend share a live, consistent view of world state with minimal bandwidth.

FEATURES

Shared state & frontend tools

Bidirectional state sync

Agent and UI share a live state object. STATE_SNAPSHOT initialises it; STATE_DELTA patches it incrementally. The frontend can also push state mutations back to the agent — both sides see the same truth.

Frontend tools

The agent can invoke tools defined in the UI — functions the browser exposes. Examples:

  • navigate(url) — agent changes the page
  • highlight(selector) — agent draws attention to an element
  • openModal(id) — agent surfaces a dialog
  • fillForm(fields) — agent pre-populates a form
Generative UI — combined, these primitives let the agent actively drive the interface, not just narrate into a chat bubble.
HUMAN-IN-THE-LOOP

Human-in-the-loop

AG-UI makes approvals and interrupts first-class citizens — not afterthoughts bolted onto a chat window.

  • Agent pauses and emits an approval event before irreversible actions (file delete, send email, deploy)
  • Frontend renders an Approve / Reject prompt; response streams back into the run
  • Mid-task human input supported — user can inject context or correct course at any step
  • Interrupt at any point — run can be paused, modified, or cancelled via the same channel
  • Audit trail — every approval decision is an event in the stream, persistent and inspectable
Safety gate — gate irreversible actions behind a human approval without changing agent logic; the protocol handles the pause-resume handshake.
Pattern — agent emits STEP_FINISHED with "requires_approval": true; frontend holds the stream; user responds; STEP_STARTED resumes the next step.

ACP — Agent Communication Protocol

REST-native agent messaging. The simplest possible contract between services-as-agents.

ACP IBM / BeeAI → Linux Foundation / REST · HTTP · async-first

What is ACP?

A REST-native open standard for agent-to-agent communication — designed to be the simplest possible contract a service-as-agent can offer.

Origin

Born at IBM Research via the BeeAI project. Now in open governance under the Linux Foundation.

Philosophy

Standardized RESTful API over plain HTTP. No SDK required. Async-first, multimodal, local-first and minimalistic.

Sweet spot

Existing REST / microservice estates that want agent capabilities without a heavy new runtime or framework dependency.

ACP

REST-native by design

  • No SDK required — any HTTP client works, including plain curl
  • Rides plain HTTP — works behind every API gateway, proxy, and load balancer already in your estate
  • Async-first — fire a run, poll or subscribe; sync and SSE streaming also supported
  • Multimodal — messages carry MIME-typed parts (text, images, files) natively
  • Local-first — discovery manifest ships with the agent package; no central registry needed
Why it matters: most enterprise services already speak REST. ACP makes those services agents with zero framework lock-in.
ACP

Core concepts

Agent
A service described by a manifest (Agent Detail). Discoverable via GET /agents/{name}. Stateless or stateful across sessions.
Run
A single invocation of an agent. Can be synchronous, asynchronous, or streamed. Has a lifecycle: created → running → completed / failed / cancelled.
Message
An ordered list of MessageParts sent to or received from an agent. Equivalent to a "turn" in the conversation.
MessagePart
Atomic content unit: a content_type (MIME), an optional name, and either inline content or a content_url.
Await
The agent pauses a run and requests additional information from the caller — ACP's elicitation / human-in-the-loop primitive.
ACP

The RESTful endpoint model

Method & PathPurpose
GET /agents List all agents available on this server — the discovery index
GET /agents/{name} Fetch the manifest (Agent Detail) for a specific agent
POST /runs Create a new run — body specifies agent name, mode (sync/async/stream), and the input message
GET /runs/{run_id} Poll run status and retrieve the output when complete
POST /runs/{run_id}/cancel Request cancellation of an in-progress run

SSE (Server-Sent Events) is used when mode: "stream" — no WebSocket dependency.

ACP

Run modes: sync · async · stream

Sync

POST /runs blocks until the agent responds. Simple request–response. Best for fast, bounded tasks where latency is acceptable.

mode: "sync"

Async (default)

POST /runs returns a run_id immediately. Caller polls GET /runs/{id} for status and result. Scales to long-running agents.

mode: "async"

Stream

POST /runs opens an SSE connection. Agent emits partial results as events — useful for token-by-token LLM output. No WebSocket needed.

mode: "stream"

Async-first design: ACP treats async as the default mental model — sync and stream are convenience overlays on top.

Discovery & manifests

Every ACP agent ships a manifest (Agent Detail) that describes its identity, capabilities, and expected message schema.

  • Offline / embedded discovery — the manifest travels with the agent package; no external registry call required at startup

  • Schema-first — callers can inspect input/output schemas before creating a run, enabling validation and tooling
  • Stateless or stateful — manifest declares whether the agent supports sessions across runs
Local-first principle: ACP is designed to work fully without cloud connectivity — package the manifest, deploy anywhere, discover locally.

ANP — Agent Network Protocol

The network layer for the open agent web. Decentralized identity and discovery across organizations.

ANP open-source / DID · JSON-LD · decentralized

The open agent web

ANP's vision: be the "HTTP of the agent era" — a universal network layer that lets agents discover, identify, and communicate across the open web without central coordination.

Internet of Agents

Just as HTTP let any server talk to any browser, ANP aspires to let any agent talk to any other agent — regardless of vendor, org, or runtime — on the open internet.

Decentralized by design

No central registry. No single authority. Agents publish crawlable description documents and carry self-sovereign identities via W3C DIDs — the web infrastructure applied to agents.

ANP

The problem: trust between strangers

Enterprise protocols (A2A, ACP) assume you already know your counterparties. But the open agent web needs agents to find and trust complete strangers.

Without ANP

  • No shared identity standard across orgs
  • No decentralized discovery — siloed registries or manual configuration
  • No way to verify an unknown agent's claims without a trusted third party
  • Cross-org collaboration requires bilateral integration agreements

With ANP

  • Self-sovereign DID identity — verifiable, no central authority
  • Crawlable description docs — find agents like web pages
  • End-to-end encrypted communication channel
  • Dynamic protocol negotiation — interop without pre-agreement
ANP

Three-layer architecture

Layer 3 — Application

Agent Description documents (JSON-LD). Publish capabilities, endpoints, and interfaces as crawlable, semantically searchable linked data.

Layer 2 — Meta-protocol

Dynamic negotiation of which protocol and data format to use — via natural language and code — so agents interoperate without pre-agreed schemas.

Layer 1 — Identity & Encryption

W3C Decentralized Identifiers (DIDs) — notably did:wba — giving each agent a self-sovereign, verifiable identity with end-to-end encrypted communication. No central authority.

ANP · Layer 1

DIDs & did:wba

W3C Decentralized Identifiers (DIDs) give each agent a globally unique, cryptographically verifiable identity that no central authority controls.

  • did:wba (Web-Based Agent) — the ANP-native DID method; resolves via HTTPS, no blockchain required
  • Self-sovereign — the agent owns its identity; no registration with a central directory
  • Verifiable — public/private key pairs let counterparties verify identity claims without a trusted third party
  • End-to-end encryption — communication channels are encrypted at the protocol layer, not just transport (TLS)
  • Portable — the same DID works regardless of hosting provider, runtime, or cloud vendor
Analogy: A DID is to an agent what a domain name + TLS certificate is to a web server — but self-issued and cryptographically bound.
ANP · Layer 2

Meta-protocol negotiation

Agents that have never met before must agree on how to talk before they can exchange meaningful content. Layer 2 handles this dynamically.

The challenge

Different agents may speak REST, JSON-RPC, GraphQL, or custom schemas. Pre-agreeing on a single format doesn't scale to an open agent internet.

The solution

Agents negotiate via natural language + code — describing their capabilities and agreeing on a protocol/format at runtime, without any shared prior configuration.

Key insight: LLM-native agents can read each other's capability descriptions in natural language and generate adapter code on the fly — making interop a runtime concern, not a design-time constraint.
ANP · Layer 3

ANP

Decentralized discovery

ANP eliminates the need for a central agent registry by treating agent discovery like web crawling — follow the links, index the semantics.

  • Crawlable linked documents — Agent Description docs reference each other via standard web links; a crawler can traverse the agent graph
  • Semantic search — JSON-LD + schema.org vocabulary means search engines (and agents) can understand capability semantics, not just keywords
  • No central registry — any agent can publish its description at its DID endpoint; no permission or registration required
  • Composable — agent descriptions can link to sub-agents, forming a navigable capability graph across the open web
  • Resilient — decentralized means no single point of failure for discovery; mirrors how DNS/web works

Aspires to enable "agent search engines" analogous to Google — but for discovering agent capabilities across organizations.

  • Agents are internal, known-party workflows — A2A or ACP are simpler
  • You need production-ready maturity today — ANP is early/research-forward
  • Your org already has a centralized identity provider (OIDC/OAuth) that covers your agent estate
  • Team lacks Web3 / DID experience and the learning curve isn't justified
Maturity note: ANP is research-forward and part of the broader "Internet of Agents" movement. Evaluate carefully before committing to production workloads.

ADJACENT PROTOCOL

AGNTCY — Internet of Agents

What it is

Open-source collective — founded by Cisco Outshift, LangChain, and Galileo — now under the Linux Foundation. Goal: build an open "Internet of Agents."

Three pillars

  • Agent Connect Protocol (ACP) — open wire protocol for agent-to-agent communication across vendors
  • OASF — Open Agentic Schema Framework; describes agent capabilities, inputs, outputs
  • Agent Directory — decentralized discovery registry for finding agents by capability

Also defines observability and identity components. Designed to collaborate with MCP & A2A, not replace them.

Name clash — ACP is ambiguous!
"ACP" refers to two unrelated protocols:

AGNTCY ACP (Agent Connect Protocol) — Cisco/LangChain/Linux Foundation open-web agent communication.

IBM ACP (Agent Communication Protocol) — IBM's agent-to-agent standard, also under the Linux Foundation.

Always qualify which ACP you mean in conversation and documentation.
ADJACENT PROTOCOL

NLWeb — the agentic web

What it is

Open project from Microsoft, led by R.V. Guha (creator of RSS and schema.org). Turns any website into a conversational, agent-queryable endpoint.

  • Uses existing structured data — schema.org markup, RSS — so no new crawling infrastructure required
  • Sites expose a natural-language query interface that agents call directly
  • Every NLWeb instance also acts as an MCP server — agents that speak MCP get NLWeb for free
  • Zero-new-infrastructure path: leverage what sites already publish
"HTML for the agentic web"

Just as HTML made the web readable by browsers, NLWeb makes it queryable by agents. The goal is a web where every page is a first-class participant in agentic workflows — without waiting for sites to rebuild their backends.

MCP synergy

Because NLWeb exposes an MCP interface, any MCP-compatible agent can query NLWeb sites out of the box — no per-site integration required.

Model-native function calling

Native tool calling handles how a model picks and invokes tools inside one runtime.

MCP standardizes the wire — how tools are described, discovered, and called across apps, servers, and vendors.

OpenAI's own Apps SDK and AgentKit are built on MCP precisely because MCP solves the cross-boundary problem that native calling does not.

  • Agents can authorize, initiate, and confirm payments on behalf of users

  • Handles delegation of spending authority + auditability
Status: emerging. AP2 and agent-commerce standards are still forming — expect flux in 2025–2026.

Consolidation signals

CROSS-CUTTING CONCERN

Observability — tracing across protocols

A single user request can traverse AG-UI → A2A → MCP across multiple agents and tool servers. Without end-to-end tracing, debugging a failed refund means archaeology across five log systems.

  • Propagate a single trace-id from the AG-UI event through every A2A task delegation and every MCP tool call
  • Log each A2A task dispatch: task ID, agent endpoint, input schema hash, and push-callback reference
  • Log every MCP tool invocation: server name, tool name, arguments (redacted), latency, and result status
  • Log every AG-UI human approval or override with timestamp and user identity
  • Emit OpenTelemetry spans at each protocol boundary — compatible with Datadog, Grafana, Jaeger
  • Capture agent reasoning tokens (summarized) alongside tool calls for explainability audits
One trace, all hops. The trace ID injected at the AG-UI layer must flow through A2A task metadata and MCP request headers unchanged — making the full causal chain reconstructable from any single hop's logs.
Approval gates are events. Every human-in-the-loop confirmation via AG-UI should emit a structured event (user ID, decision, timestamp, context hash) — not just a log line — so it can be joined to the trace.
Watch for fan-out. A2A parallel task dispatch multiplies span count rapidly. Set sampling budgets by trace root, not per-span, to avoid log volume explosions in high-throughput pipelines.
CROSS-CUTTING CONCERN

Security & Trust — identity at every hop

User → Agent AG-UI

App session token identifies the human user. AG-UI events carry session context; approval actions are signed with user identity. Frontend never has direct tool access.

  • Session token scoped to app; short-lived
  • Human approval events signed + timestamped
  • Agent cannot bypass the UI gate

Agent → Agent A2A

mTLS between agent runtimes; OAuth client credentials scoped to the delegating agent's identity. The receiving specialist agent enforces its own authorization — it does not trust the orchestrator blindly.

Agent → Tools MCP

OAuth 2.0 scopes grant least-privilege access per agent role. MCP servers enforce scope boundaries; no agent can call a tool outside its granted scopes, regardless of what the orchestrator requested.

Discovery ≠ authorization. ANP resolves and cryptographically verifies an external agent's identity via DIDs — but that only establishes who it is. Business logic, spend limits, data-access policies, and approval gates must still be enforced inside your own agent. Trust is graduated, not binary.
CROSS-CUTTING CONCERN

Cost, Latency & Failure Modes

Each protocol hop adds latency and spend. Design budgets, degradation paths, and failure contracts before scaling agents.

Latency & Cost Budget

Every A2A delegation and MCP tool call adds round-trip latency and LLM token spend. Set per-request budgets:

  • Max hop depth — prevent runaway recursion in orchestrators
  • Max parallel tasks — A2A fan-out can multiply costs 10×
  • Token ceilings — per agent, per session, per tool call
  • Timeout escalation — short timeout → push callback → SLA alert

Long-Running Task Handling

A2A push model is designed for tasks that outlast HTTP timeouts — but partial failure must be handled explicitly:

  • Idempotent task IDs — safe to retry on transient failure
  • Checkpoint artifacts — partial results preserved on failure
  • Dead-letter handling — tasks with no push callback after TTL
  • Compensation tasks — rollback logic for committed side effects

Graceful Degradation

When a downstream tool server or specialist agent is unavailable, the system should degrade gracefully — not fail completely:

  • Fallback responses — cached data or reduced-capability mode
  • Circuit breakers — stop hammering unavailable MCP servers
  • Human escalation — AG-UI surfaces failures for manual override
  • Partial results — return what's available; flag what's missing

Hard-won guidance

Anti-patterns to avoid

⚠️ Protocol sprawl

Don't adopt all six on day one. Each adds attack surface & ops burden. Earn each layer.

⚠️ A2A as an API bus

A2A is for agents, not a fancy wrapper over a REST call — that's MCP or plain HTTP.

⚠️ Trusting Agent Cards blindly

Discovery ≠ authorization. Verify identity (ANP / mTLS) & scope every capability.

🔐 Tool-poisoning & injection

MCP tool descriptions are untrusted input. Pin servers, review, sandbox.

🔐 Over-broad scopes

One mega-token for everything = blast radius. OAuth 2.1 scope per server; least privilege.

🌀 Unbounded agent loops

Agent-to-agent recursion can fan out forever. Budget hops, depth & spend.

Cross-cutting

A governance checklist for every layer

Identity & access

  • OAuth 2.1 + scopes on every remote MCP server
  • Per-skill auth schemes declared in A2A Agent Cards
  • DID / mTLS verification before trusting external agents
  • Short-lived, audience-bound tokens — no static keys

Observability
