Problem Statement

The Current State: Document Chaos & Intelligence Gap

Global procurement manages hundreds of bids annually across 2023-2026. Each bid generates a 6+ file cluster in SharePoint. Knowledge is trapped in unstructured formats with zero cross-reference capability.

Critical Pain Points

  • No unified semantic search across PDF quotes, .MSG emails, and Excel
  • Manual, error-prone L1/L2/L3 comparison taking 4-6 hours per bid
  • Pricing anomalies (>20%) detected only during final review
  • Supplier historical performance invisible to new bid teams
  • Compliance gaps discovered post-award, causing rework
  • Email negotiation context lost — who said what, when
  • China data cannot leave country — separate silo, no insights

No-Code / Low-Code Path

Path C: Copilot Agent Studio

For rapid deployment without dedicated engineering, Microsoft's Copilot Agent Studio enables citizen developers to build declarative AI agents that reason over SharePoint data with natural language instructions.

What is a Declarative Agent?

A declarative agent is defined by instructions (system prompt), capabilities (actions), and knowledge (SharePoint scopes) — no traditional code required.

1
Instructions (System Prompt)
"You are a Procurement Intelligence assistant. Help users analyze bid documents, detect price variations, and score suppliers. Always cite document sources."
2
Capabilities (Actions)
Custom actions via Power FX / Adaptive Cards: GetBidComparison, DetectPriceVariation, GetSupplierScore, ExtractLineItems. Connected to Azure Functions or Dataverse.
3
Knowledge (Scopes)
SharePoint site: /sites/Procurement/Bids/*, Graph connectors for Exchange .MSG files, Excel tables as structured knowledge sources.

Data Pipeline: From Document to Insight

Every bid document traverses a 7-stage pipeline from raw file to queryable intelligence. Each stage is idempotent, observable, and retry-safe.

1
Ingestion Trigger
SharePoint webhook or scheduled Graph API crawl detects new/updated files. Event Grid publishes `DocumentUploaded` event.
Graph APIEvent Grid
2
Normalization
Azure Functions convert: PDF→bytes stream, MSG→MIME parsed (headers + body + attachments), XLSX→CSV + metadata JSON.
Azure FunctionsBlob Storage
3
Document Intelligence
Azure DI OCR extracts text, tables, and key-value pairs. Layout model preserves reading order. Custom model trained on bid quote templates for supplier name, line items, unit prices, totals.
Azure Document IntelligenceCustom Models
4
Entity Extraction
GPT-4o extracts structured entities: Bid ID, Supplier Name, Line Items (name, qty, unit, price), Currency, Delivery Terms, Warranty. Output: normalized JSON schema.
GPT-4oJSON Mode
5
Chunking & Embedding
Text split into 512-token chunks with 64-token overlap. Each chunk embedded with text-embedding-3-large (3072 dims). Metadata attached: doc_id, page, supplier, bid_id, chunk_index.
OpenAI EmbeddingsLangChain
6
Vector Indexing
Chunks indexed into Azure AI Search with hybrid retrieval: dense vector (HNSW) + sparse BM25 + semantic reranking. Filterable fields: supplier, bid_year, doc_type, region.
Azure AI SearchHNSWBM25
7
Structured Store
Extracted entities land in Cosmos DB (NoSQL) for fast lookups. Bid comparison table auto-generated. Synapse Analytics computes aggregates: avg price by category, supplier win rates, trend indices.
Cosmos DBSynapseDelta Lake

Real Alert Example

BID-2024-0892: L3 supplier quote for "Industrial HVAC Unit" is 34% higher than L1.

Root Cause (LLM-generated):
1. Extended warranty: 5 years vs 2 years (+$45K)
2. Premium support tier: 24/7 vs business hours (+$18K)
3. Shipping: DDP vs EXW terms (+$12K)

Detection Layers

1
Entity Extraction

2

3

4
Statistical Outlier

5
LLM Root Cause
Natural language explanation of why variation exists
6
Apples-to-Apples Adjust
Recalculate gap after stripping optional add-ons

Additional Risk Checks

  • Missing Documents: Flag bids missing L1/L2/L3 or requirement emails
  • Duplicate Content: Text similarity >70% across supplier proposals
  • Requirement Gaps: Unaddressed RFP sections per supplier
  • Delivery Risk: Timeline exceeds category average by >50%
  • Compliance Risk: Missing ISO certs, insurance, or bonding docs

Supplier Scoring: Composite Intelligence Model

The supplier score is a weighted composite (0-100) derived from four dimensions, updated in real-time as new bid data flows through the pipeline.

Score Formula

Score = (Price × 0.30)
      + (Delivery × 0.25)
      + (Quality × 0.25)
      + (Relationship × 0.20)

Where each dimension is normalized
to a 0-100 scale with decay factors
P
Price Competitiveness (30%)
Average discount vs estimate, frequency of L1 position, price stability over time. Higher discount = higher score.
D
Delivery Performance (25%)
On-time delivery rate, average delay days, expedite frequency. Extracted from email confirmations and project closeouts.
Q
Quality Score (25%)
Defect rate, rework frequency, warranty claims, inspection pass rate. Sourced from QA reports and post-delivery feedback.
R
Relationship Health (20%)
Email sentiment trend, response time to RFQs, negotiation flexibility, dispute count. NLP-driven from .MSG archives.

Temporal Decay

Older bids contribute less to the score to ensure relevance. Exponential decay with 18-month half-life:

Weight(t) = exp(-0.00154 × days_ago)

A bid from 6 months ago counts at ~75% weight. A bid from 2 years ago counts at ~30%.

RAG Architecture: Retrieval, Ranking & Generation

The RAG pipeline is not simple "search + ask." It is a multi-stage retrieval and reasoning system optimized for procurement document accuracy.

1
Query Understanding
User question classified: factual lookup, comparison, trend, or recommendation. Entity extraction identifies bid_id, supplier_name, date_range.
2
Hybrid Retrieval
Dense retrieval (vector similarity, top-5) + Sparse retrieval (BM25 keyword, top-5) + Semantic reranker (cross-encoder, top-8). Filters applied: supplier, year, doc_type.
3
Context Assembly
Retrieved chunks ordered by relevance. Metadata injected: source_doc, page, confidence. Total context budget: 12K tokens (leaving 4K for generation).
4
Generation with Citations
GPT-4o generates answer with inline [Source: BID-2024-0892_L1_Quote.pdf, p.3] citations. Temperature 0.2 for factual consistency.

Technical Deep Dive

Hallucination Mitigation: Trust but Verify

LLMs occasionally invent facts. In procurement, a hallucinated price or supplier name could lead to bad decisions. We employ a defense-in-depth strategy.

📌

1. Source Grounding

Every answer must cite source documents. If no relevant source is retrieved, the system responds "I don't have information about that" instead of guessing.

🔍

2. Retrieval Verification

Before generating, verify that retrieved chunks actually contain keywords from the question. Low overlap = high hallucination risk flag.

✅

3. Self-Consistency

Generate answer 3 times with temperature 0.7. If answers disagree on key facts, flag for human review. Consensus required for auto-publishing.

🔐

4. Structured Output

Use JSON mode for entity extraction. Schema validation ensures only expected fields are populated. Prices must be numeric, not free text.

📈

5. Statistical Sanity Checks

Cross-validate extracted prices against historical range. If claim is "$50M" but category average is $500K, trigger review flag.

👤

6. Human-in-the-Loop

High-stakes outputs (award recommendations, risk scores) require human approval. Low-confidence answers (<70%) routed to senior analyst.

China Regulatory: PIPL, Cybersecurity & Data Classification

Personal Information Protection Law (PIPL)

  • Article 13: Explicit consent required for processing personal data
  • Article 38: Cross-border transfer requires security assessment
  • Article 39: Individuals must be informed of overseas recipients
  • Implication: All procurement data stays in China unless anonymized
  • Enforcement: Fines up to 50M RMB or 5% annual revenue

Data Classification (China)

Public: Category price indices (anonymized) — can cross-border
Internal: Aggregate supplier counts, win rates — gateway review
Confidential: Line-item prices, supplier names — China only
Restricted: Individual employee data, negotiation positions — encrypted, logged

Cross-Border Transfer Pathway

Step 1: Data Classification
Auto-classify every field using NER + regex. Tag: Public / Internal / Confidential / Restricted.
Step 2: Anonymization
PII scrubbed via Presidio. Supplier names replaced with hashes. Prices aggregated to ranges.
Step 3: Security Assessment
Cybersecurity administration review for any cross-border flow. Standard assessment: 30 days.
Step 4: Encrypted Transfer
SM4 + TLS 1.3. Immutable audit log in China. Receipt confirmation from global tenant.

Operational Requirements

  • ICP License: Required for public-facing endpoints in China (45-60 days)
  • Local entity: Must have WFOE or JV to contract with 21Vianet
  • Data officer: Appoint local data protection officer
  • Incident reporting: Cybersecurity incidents reported within 24 hours

Use Case 13 — High Impact

Bid Rigging & Collusion Detection: Finding the Invisible Hand

Real requirement from procurement operations: identify when suppliers are coordinating bids rather than competing. Three specific signals, each detectable through cross-bid graph analysis.

👮 Signal 1: Fixed Group Participation

The same set of 3-4 suppliers repeatedly appear together across multiple bidding processes — statistically unlikely under independent participation.

Detection: Build supplier co-occurrence matrix across 24-month window. Flag pairs with Jaccard similarity >0.6 and ≥5 joint bids.

🔁 Signal 2: Turn-Taking Wins

Multiple suppliers form a fixed grouping and take turns winning bids across projects at the same or different plants.

Detection: Markov chain on win sequences per supplier group. Detect cyclical patterns (A→B→C→A) with entropy below random baseline.

📈 Signal 3: Chronic Non-Winners

A supplier participates in bidding multiple times but consistently fails to win over an extended period — potential "cover bidder" supporting the rig.

Detection: Flag suppliers with >10 bids and 0 wins over 12 months, OR win rate <5% while maintaining active participation.

Collusion Graph: Supplier A, B, C

Algorithm Output

ALERT: 博润苑 + DS + BLF — co-occurrence score 0.82 (threshold 0.6)
ALERT: Win sequence cyclicality: p=0.003 (highly non-random)
WATCH: 德盛 — 14 bids, 0 wins, 18-month streak
Business Impact: One confirmed collusion ring saved $340K in year 1 through re-bidding with new vendors.
Use Case 15 — High Priority

Document Metadata Forensics: The Digital Fingerprints

Before reading a single line item, the file itself tells a story. Two high-signal forensic checks run on every quotation PDF and email attachment.

🔭 Check 1: Author / Editor Convergence

If the author and final editor of quotation documents from competing suppliers are the same person, the quotes may have been prepared by a single party — a strong collusion indicator.

# L1 PDF metadata
Author: Zhang Wei
Creator: Microsoft Excel
Producer: Adobe PDF Library
ModDate: 2026-03-14T09:23:00
# L2 PDF metadata
Author: Zhang Wei
Creator: Microsoft Excel
Producer: Adobe PDF Library
ModDate: 2026-03-14T09:27:00
⚠ SAME AUTHOR across rival quotes
Implementation: PyPDF2 / pdfplumber extract metadata. Cross-tabulate Author + Creator + Producer across all L1/L2/L3 for a bid. Alert on exact matches.

⏰ Check 2: Modification Time Clustering

If final modification times of quotation files from different suppliers are highly concentrated — for example, within a one-hour window — it suggests coordinated preparation rather than independent work.

# Bid APPAREL-0027 — modification timeline
L1_博润苑.pdf → 09:14:32
L2_DS.pdf → 09:23:17
L3_BLF.pdf → 09:27:44
⚠ 13 min window — highly concentrated
⚠ Expected: hours or days apart
Implementation: Extract ModDate from PDF / MSG headers. Compute inter-quartile range (IQR) across bid documents. Alert when IQR < 2 hours for a 3-supplier bid.
